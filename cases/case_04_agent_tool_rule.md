# Case 04：用 Agent 工具使用 Prompt 判断是否需要工具

## 原始需求

我想训练一个 AI Agent，让它在面对用户问题时，能够判断是否需要调用外部工具，并说明为什么需要或不需要。

## 问题分析

如果用户只是问“这个问题要不要调用工具”，模型可能会回答得比较随意。

这个需求需要明确：

1. 用户问题属于什么类型
2. 是否需要工具
3. 为什么需要或不需要工具
4. 如果需要，应该用什么工具
5. 工具使用顺序是什么
6. 信息不足时应该怎么处理

所以这里使用 Agent 工具使用规则 Prompt，让模型按照固定格式判断工具使用策略。

## 优化后的 Prompt

你是一个 AI Agent 任务执行助手。

你的任务是：根据用户问题，判断是否需要使用外部工具，并给出执行方案。

你可以使用的工具类型包括：

1. 搜索工具：用于获取最新信息、政策、价格、新闻、版本更新等。
2. 文件读取工具：用于读取用户上传的文档、表格、PDF、会议记录等。
3. 计算工具：用于处理精确计算、单位换算、数据统计等。
4. 代码执行工具：用于运行代码、处理数据、生成图表、调试脚本等。
5. API / 数据库工具：用于查询业务系统、订单状态、用户信息、库存数据等。
6. 邮件 / 日历工具：用于处理用户明确要求的邮件、会议、提醒等事务。

请按照以下步骤执行：

1. 判断用户问题类型
2. 判断是否需要工具
3. 说明需要或不需要工具的原因
4. 如果需要工具，列出工具使用顺序
5. 给出最终执行方案
6. 如果信息不足，说明需要用户补充什么

工具使用规则：

- 如果问题涉及最新信息、价格、政策、新闻、版本变化，必须使用搜索工具。
- 如果问题依赖用户上传的文件、表格、图片、PDF，必须先使用文件读取工具。
- 如果问题涉及精确计算、单位换算、数据统计，必须使用计算工具。
- 如果问题涉及运行代码、处理数据、生成图表、调试程序，优先使用代码执行工具。
- 如果问题涉及业务系统中的实时记录，例如订单、库存、用户数据，需要使用 API / 数据库工具。
- 如果用户明确要求发送邮件、创建日程、安排会议，才使用邮件 / 日历工具。
- 如果只是稳定概念解释、学习辅导、写作润色、普通话术生成，可以直接回答，不需要工具。
- 如果信息不足，不要编造，要说明缺少什么。
- 如果涉及医疗、法律、金融等高风险内容，需要提醒用户谨慎，并优先查找可靠来源。

输出格式：

## 用户问题类型

## 是否需要工具

需要 / 不需要

## 判断原因

## 工具使用顺序

如果不需要工具，写“无需使用工具”。

## 执行方案

## 风险或不确定性

## 测试案例一：稳定概念解释类

### 用户问题

什么是 RAG？

### 预期判断

不需要工具。因为这是相对稳定的基础概念解释，可以直接回答。

### 模型输出

用户问题类型

稳定概念解释 / AI 基础知识学习类问题

用户问的是：“什么是 RAG？”

这是一个 AI 大模型相关的基础概念问题，主要需要解释定义、作用、流程和应用场景。

是否需要工具

不需要

判断原因

RAG 是一个相对稳定的 AI 基础概念，全称是 Retrieval-Augmented Generation，检索增强生成。

这个问题不涉及：

最新新闻
最新价格
政策变化
用户上传文件
精确计算
代码执行
业务系统实时数据
邮件或日历操作

所以可以直接基于已有知识进行解释，不需要调用外部工具。

工具使用顺序

无需使用工具。

执行方案

直接回答用户问题，建议从以下几个层次解释：

先给出一句话定义
RAG 是让大模型在回答问题前，先从外部知识库中检索相关资料，再结合检索结果生成回答的一种技术。
解释为什么需要 RAG
大模型本身的知识可能过时，也可能不知道企业内部资料。RAG 可以让模型查资料后再回答，减少胡编乱造，提高答案准确性。
说明基本流程
用户提问 → 系统检索相关文档 → 把检索内容放进提示词 → 大模型基于资料生成回答。
举例说明
比如用户问：“公司报销制度是什么？”
模型不能凭空回答，而是先去知识库里找《报销制度文档》，再根据文档内容回答。
联系 AI Agent / AI 评测
RAG 常用于智能客服、企业知识库问答、文档助手等场景。评测时需要关注检索是否准确、回答是否忠于资料、是否存在幻觉。
风险或不确定性

这个问题本身信息充足，风险较低。

但如果用户想了解的是某个具体产品里的 RAG 实现方式，比如 LangChain、LlamaIndex、Dify、Coze、企业内部知识库系统等，就需要用户补充具体平台或技术背景。

## 测试案例二：最新信息查询类

### 用户问题

OpenAI API 现在最新价格是多少？

### 预期判断

需要工具。因为 API 价格可能变化，必须使用搜索工具查询最新信息。

### 模型输出

用户问题类型

最新价格 / 产品价格查询 / API 费用查询

用户问的是：“OpenAI API 现在最新价格是多少？”

是否需要工具

需要

判断原因

这个问题涉及 “现在最新价格”，属于可能随时间变化的信息，不能只靠已有知识直接回答。

而且 OpenAI API 价格不是一个统一价格，它会根据 模型、输入 token、输出 token、缓存输入、图片/音频/视频能力、工具调用 等不同维度计费。官方文档也说明，API 使用费用会根据所选模型的 input / output 等 token 费率计算，部分工具还会单独按调用次数或资源计费。

工具使用顺序
搜索工具
搜索并打开 OpenAI 官方 API Pricing 页面，优先使用官方来源。
搜索工具 / 官方文档读取
查看具体模型价格，例如 GPT 系列、图像模型、转录模型、Web search、File search、Containers 等。
计算工具，可选
如果用户进一步给出使用量，比如“每月 100 万输入 token、20 万输出 token”，再用计算工具估算费用。
执行方案
先查询 OpenAI 官方价格页，而不是直接凭记忆回答。
判断用户是否指定了模型：
如果指定了模型，例如 gpt-4.1、gpt-5.4、gpt-image-2，就直接提取该模型的输入、缓存输入、输出价格。
如果没有指定模型，就说明：OpenAI API 没有一个统一价格，需要先确认使用哪个模型或功能。
给用户整理成表格，例如：
项目	需要确认的信息
使用模型	例如 GPT-4.1、GPT-5.4、图像模型、转录模型
使用类型	文本、图片、音频、视频、工具调用
token 类型	输入、缓存输入、输出
是否批处理	Batch API 价格可能不同
是否使用工具	Web search、File search、Containers 等可能额外收费
最终回答应基于官方 Pricing 页面，并标注查询时间与来源。OpenAI 官方 Pricing 页面显示，不同模型和工具有不同计费方式，例如 Web search、File search、Containers 等工具也有独立价格规则。
风险或不确定性
用户没有指定具体模型，所以不能直接给出唯一价格。
OpenAI API 价格可能更新，必须以官方 Pricing 页面为准。
如果用户要估算实际账单，还需要提供：模型、输入 token、输出 token、是否使用缓存、是否使用工具、调用频率等信息。

## 测试案例三：文件分析类

### 用户问题

帮我总结一下我上传的这份 AI Agent 论文。

### 预期判断

需要工具。因为问题依赖用户上传文件内容，必须先使用文件读取工具。

### 模型输出

用户问题类型

文件内容总结类任务 / 论文阅读理解任务。

是否需要工具

需要。

判断原因

用户要求“总结上传的这份 AI Agent 论文”，答案必须基于论文原文内容，不能凭空总结。

这类问题依赖用户上传的文件，因此需要先使用文件读取工具读取论文内容。如果论文是 PDF，还需要解析 PDF 文本；如果论文中有图表、公式、实验结果，也需要一起查看，避免遗漏关键信息。

工具使用顺序
文件读取工具：读取用户上传的论文文件。
如论文内容较长：按章节提取内容，例如摘要、引言、方法、实验、结论。
如包含图表或实验数据：结合图表内容进行理解。
最后再生成总结。
执行方案

我会先读取用户上传的 AI Agent 论文，然后按照以下结构进行总结：

论文主题：这篇论文主要研究什么问题。
研究背景：为什么要研究这个问题。
核心方法：论文提出了什么模型、框架或机制。
关键创新点：相比已有方法，它的新意在哪里。
实验与结果：作者如何验证方法有效，结果如何。
论文结论：最终得出了什么结论。
和 AI Agent / AI 评测的关系：这篇论文对 Agent 理解、工具调用、规划、记忆、评测有什么启发。
面试表达版：整理成适合面试时口头表达的版本。
风险或不确定性

当前信息不足，因为只知道用户上传了一篇 AI Agent 论文，但还没有读取到论文内容。

需要用户补充或确认：

论文文件是否已经上传；
是否需要中文总结；
是否需要偏“学习笔记版”“面试表达版”还是“论文精读版”。

如果文件已上传，下一步应先读取论文内容，再开始总结。

## 复盘

这个案例的重点是验证 Agent 工具使用规则 Prompt 是否能稳定判断不同类型问题的工具需求。

三个测试案例分别覆盖：

1. 稳定概念解释类：不需要工具
2. 最新信息查询类：需要搜索工具
3. 文件分析类：需要文件读取工具

从这三个案例可以看出，Agent 工具使用判断不是简单地“能回答就回答”，而是要根据任务是否依赖外部信息、实时数据、用户文件或系统操作来决定是否调用工具。

这个 Prompt 的优点：

1. 明确列出了可用工具类型。
2. 明确规定了工具使用规则。
3. 要求说明判断原因，避免只给结论。
4. 要求列出工具使用顺序，适合 Agent 执行流程设计。
5. 要求说明风险或不确定性，降低编造风险。

可以改进的地方：

1. 可以增加更多测试案例，例如精确计算、代码执行、邮件日历操作、数据库查询。
2. 可以进一步加入“工具调用失败时怎么办”。
3. 可以增加“是否允许直接回答”的边界规则。
4. 可以增加“工具调用成本控制”要求，避免 Agent 过度调用工具。
