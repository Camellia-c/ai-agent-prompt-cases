# Prompt 评分标准与评分方式分类

## 1. 文件用途

这个文件用于建立一套 Prompt 质量评估标准。

它主要回答五个问题：

1. 评什么：Prompt 质量应该从哪些维度判断？
2. 怎么打分：不同分数代表什么质量等级？
3. 谁来评：评分方式有哪些？
4. 怎么选：不同任务应该选择哪种评分方式？
5. 怎么分类：Prompt 可以分成哪些类型？

这个文件的目标不是判断 Prompt 写得长不长、看起来高级不高级，而是判断：

- 用户需求是否被准确转化
- 任务目标是否清楚
- 输出是否可控
- 模型是否容易稳定执行
- 是否能减少跑偏、遗漏和编造
- 是否适合复用和迭代优化

简单说：

> Prompt 评分标准 = 判断一个 Prompt 好不好、差在哪里、怎么优化的依据。

---

## 2. Prompt 评分维度：评什么？

| 维度 | 说明 | 低质量表现 | 高质量表现 |
|---|---|---|---|
| 目标清晰度 | 是否明确告诉模型要完成什么任务 | 只说“帮我看看”“总结一下” | 明确说明要分析什么、生成什么、解决什么问题 |
| 角色设定 | 是否告诉模型以什么身份完成任务 | 没有角色，模型自由发挥 | 明确角色，如学习助手、评测员、Agent 执行助手 |
| 背景完整性 | 是否说明面向谁、使用场景、目的 | 不知道给谁看、用于什么场景 | 明确用户对象、业务背景、使用目的 |
| 输入明确性 | 是否说明模型要处理什么材料 | 输入内容不清楚或缺少边界 | 明确给出材料、字段、变量或粘贴位置 |
| 执行步骤 | 是否规定模型的处理流程 | 模型想到哪写到哪 | 明确先判断、再分析、再输出 |
| 输出格式 | 是否规定回答结构 | 输出散乱，不方便使用 | 明确要求表格、清单、标题、评分表等格式 |
| 质量标准 | 是否说明什么样的回答算好 | 只要求“写好一点” | 明确要求准确、完整、口语化、适合面试等 |
| 边界控制 | 是否说明不能做什么、信息不足怎么办 | 容易编造，不确定也强答 | 明确要求不编造、说明缺失信息、标注不确定性 |
| 可复用性 | 是否能替换变量后反复使用 | 只能用于一次具体问题 | 可以替换主题、资料、用户问题后继续使用 |
| 可评估性 | 是否方便后续评估输出并改进 | 没有评分或复盘依据 | 能根据输出判断问题并继续优化 Prompt |

---

## 3. 5 分制评分等级：怎么打分？

| 分数 | 质量等级 | 说明 |
|---:|---|---|
| 1 分 | 很差 | 需求非常模糊，没有角色、格式、标准，输出高度不可控 |
| 2 分 | 较差 | 有基本任务，但缺少背景、步骤、输出格式或质量要求 |
| 3 分 | 一般 | 任务比较清楚，有基本结构，但约束不完整，输出稳定性一般 |
| 4 分 | 较好 | 角色、目标、步骤、格式、质量要求较完整，能稳定完成大多数任务 |
| 5 分 | 优秀 | 目标清晰、结构完整、边界明确、可复用、可评估、方便迭代优化 |

## 4. 评分方式分类：谁来评、怎么评？

Prompt 或模型输出的评分方式，主要可以分为四类：

1. 人工评分
2. 大模型评分
3. 规则 / 程序评分
4. 混合评分

---

### 4.1 人工评分 Human Evaluation

人工评分是由真实评测员根据评分标准进行判断。

适合评估主观性强、需要理解语境和业务场景的内容，例如：

- 回答是否真正解决了用户问题
- 语气是否自然
- 是否符合业务场景
- 解释是否容易理解
- 是否有隐性误导
- 用户体验是否好

优点：

- 判断灵活
- 更能理解真实用户场景
- 适合复杂、开放、主观任务

缺点：

- 成本高
- 速度慢
- 不同评测员之间标准可能不一致
- 大规模评测很难完全依赖人工

适合场景：

- 客服话术质量评估
- 教育类内容表达评估
- 开放式问答质量判断
- 高风险样本复核
- Prompt 优化结果人工验收

面试表达：

> 人工评分适合处理主观性强、场景复杂、需要语境判断的任务，比如语气、完整性、用户体验等。但它成本高、效率低，而且不同评测员之间可能存在标准不一致，所以需要配合清晰的评分 Rubric。

---

### 4.2 大模型评分 LLM-as-a-Judge

大模型评分，也叫 LLM-as-a-Judge，是让大模型根据评分标准对 Prompt 或模型输出进行自动评估。

它通常会使用一个评测 Prompt，让大模型输出：

- 总体评分
- 分项评分
- 问题说明
- 修改建议
- 更好的回答版本

适合评估：

- 回答是否相关
- 内容是否完整
- 表达是否清晰
- 是否符合指定格式
- 两个 Prompt 哪个更好
- 输出是否满足任务要求

优点：

- 速度快
- 成本低于人工
- 可以批量处理
- 能给出理由和修改建议
- 适合初筛和辅助评审

缺点：

- 评分可能不稳定
- 可能偏向更长、更流畅的回答
- 可能漏掉细微事实错误
- 可能被表面合理但实际错误的内容误导
- 评测 Prompt 写得不好，评分结果也会差

适合场景：

- 批量评估模型回答
- Prompt A/B 对比
- 输出质量初筛
- 自动生成修改建议
- 辅助人工评审

面试表达：

> 大模型评分，也叫 LLM-as-a-Judge，是用大模型根据评分标准对回答进行自动评估。它适合批量评测、初筛和生成问题分析，但它本身也可能不稳定或产生偏差，所以通常需要人工抽检或和规则评分结合使用。

---

### 4.3 规则 / 程序评分 Rule-based / Programmatic Evaluation

规则 / 程序评分是通过明确规则、代码或脚本进行自动判断。

它不依赖人的主观判断，也不主要依赖大模型理解，而是用明确条件判断是否达标。

常见规则包括：

- 是否输出 JSON 格式
- 是否包含指定字段
- 是否在规定字数范围内
- 是否包含关键词
- 是否命中正确分类标签
- 数学计算结果是否正确
- 是否出现禁止词
- 是否引用来源
- 是否调用了指定工具

适合评估：

| 评测内容 | 示例 |
|---|---|
| 格式是否正确 | 是否输出 JSON、Markdown 表格 |
| 字段是否完整 | 是否包含 title、summary、score |
| 字数是否符合要求 | 是否在 100-200 字之间 |
| 关键词是否出现 | 是否提到 RAG、Embedding、检索 |
| 标准答案是否正确 | 计算结果是否等于标准答案 |
| 分类是否正确 | 是否选中正确标签 |
| 安全规则是否触发 | 是否出现禁止词或高风险内容 |
| 工具调用是否正确 | 是否调用了搜索、文件读取、计算工具 |

优点：

- 稳定
- 可重复
- 速度快
- 成本低
- 适合自动化
- 不容易受主观偏好影响

缺点：

- 不够灵活
- 很难判断语义质量
- 容易只看形式，不看内容
- 对开放式回答支持较弱
- 规则设计不好会误判

适合场景：

- 格式检查
- 字段完整性检查
- 关键词检查
- 分类任务
- 标准答案任务
- 工具调用日志检查
- 批量自动化评测

面试表达：

> 规则评分适合评格式、字段、关键词、字数、分类标签、标准答案等客观指标。它稳定、便宜、适合自动化，但不擅长判断开放式回答的语义质量，所以通常不能单独用于复杂文本评测。

---

### 4.4 混合评分 Hybrid Evaluation

混合评分是将人工评分、大模型评分和规则 / 程序评分结合起来。

真实业务里，最常见的不是只用一种评分方式，而是多种方式配合使用。

常见流程：

规则评分先检查格式和硬性要求  
↓  
大模型评分判断内容质量  
↓  
人工抽检高风险、争议或低置信度样本

适合评估：

- RAG 问答质量
- AI Agent 工具调用过程
- 客服质检
- 内容安全
- 多轮对话质量
- Prompt 优化前后效果
- 复杂业务场景下的模型输出质量

优点：

- 兼顾效率和质量
- 规则评分保证基础格式和硬性要求
- 大模型评分提高覆盖率
- 人工评分兜底复杂判断
- 更接近真实业务流程

缺点：

- 设计成本更高
- 需要明确每种评分方式负责什么
- 需要处理不同评分结果不一致的情况
- 需要设计抽检机制和仲裁机制

适合场景：

- 大规模评测
- 高风险内容评估
- 复杂 Agent 任务评测
- RAG 问答评测
- 模型上线前验收

面试表达：

> 实际业务里我更倾向于混合评分。比如先用规则评分检查格式和硬性要求，再用大模型评分判断内容质量，最后由人工抽检高风险样本或争议样本。这样可以兼顾效率、成本和准确性。

---

## 5. 不同评分方式怎么选？

选择评分方式时，核心看三个问题：

1. 任务是否客观？
2. 是否需要理解语义和语境？
3. 是否需要大规模、低成本、自动化处理？

如果任务有标准答案、明确格式、固定字段，优先考虑规则 / 程序评分。

如果任务是开放式回答，需要判断语义、完整性、表达质量，可以使用大模型评分，并配合人工抽检。

如果任务主观性强、风险高、业务语境复杂，需要人工评分兜底。

如果任务复杂且规模较大，通常使用混合评分。

| 任务类型 | 更适合的评分方式 | 原因 |
|---|---|---|
| 判断 JSON 格式是否正确 | 规则评分 | 格式是否正确可以用程序稳定判断 |
| 判断字段是否完整 | 规则评分 | 可以检查是否包含指定字段 |
| 判断数学计算是否正确 | 规则评分 / 程序评分 | 有明确标准答案 |
| 判断分类标签是否正确 | 规则评分 / 人工抽检 | 有标签标准，但边界样本可能需要人工 |
| 判断回答是否完整 | 大模型评分 + 人工抽检 | 需要理解语义和任务要求 |
| 判断回答是否自然 | 人工评分 / 大模型评分 | 涉及主观体验和表达质量 |
| 判断客服话术是否合适 | 人工评分 + 大模型评分 | 需要结合语气、场景和用户感受 |
| 判断 RAG 回答是否基于资料 | 大模型评分 + 规则评分 + 人工抽检 | 既要看引用和资料一致性，也要看语义质量 |
| 判断 Agent 是否正确调用工具 | 规则评分 + 大模型评分 | 工具调用日志可以规则检查，调用理由可以模型判断 |
| 判断 Prompt A 和 Prompt B 哪个更好 | 大模型评分 + 人工复核 | 可以先自动对比，再人工确认 |
| 大规模批量评估 | 规则评分 + 大模型评分 | 需要效率和成本控制 |
| 高风险内容评估 | 规则评分 + 大模型评分 + 人工兜底 | 需要降低漏判风险 |

### 选择原则总结

| 情况 | 优先选择 |
|---|---|
| 有明确标准答案 | 规则 / 程序评分 |
| 有明确格式要求 | 规则 / 程序评分 |
| 需要判断语义质量 | 大模型评分 |
| 需要判断用户体验 | 人工评分 / 大模型评分 |
| 主观性很强 | 人工评分 |
| 风险较高 | 混合评分 |
| 样本量很大 | 规则评分 + 大模型评分 |
| 需要上线验收 | 混合评分 |

## 6. Prompt 类型分类

不同类型的 Prompt，设计重点和评分重点不一样。

---

### 6.1 学习型 Prompt

用于学习一个概念或知识点。

典型任务：

- 解释一个 AI 概念
- 生成学习笔记
- 输出面试表达
- 给出例子和自测题

设计重点：

- 面向对象清楚
- 解释通俗
- 专业词要解释
- 有结构化输出
- 能转化成面试表达

评分重点：

- 是否解释清楚
- 是否适合目标学习者
- 是否有例子
- 是否能用于复习或面试表达
- 是否避免堆术语

示例：

请面向 AI 评测岗位学习者解释 RAG。  
请从定义、解决的问题、基本流程、应用场景、面试问法五个方面输出。  
要求语言通俗，专业词需要解释。

---

### 6.2 总结型 Prompt

用于把资料、文章、会议记录、论文内容整理成结构化结果。

典型任务：

- 总结资料
- 提炼重点
- 整理流程
- 生成复习笔记
- 输出行动项
- 整理会议纪要
- 提取论文核心观点

设计重点：

- 明确输入材料
- 明确总结目标
- 明确输出结构
- 不照抄原文
- 区分原文信息和模型推理补充
- 信息不足时说明缺口

评分重点：

- 是否覆盖原文重点
- 是否结构清晰
- 是否存在明显遗漏
- 是否有自行编造
- 是否区分资料已有信息和模型补充信息
- 是否适合后续复习、汇报或面试表达

示例：

请把下面资料整理成适合 AI 评测岗位学习者复习的笔记。  
要求包括核心结论、关键概念、流程机制、面试问法和 1 分钟表达。  
如果资料信息不足，请明确说明，不要自行编造。

---

### 6.3 评测型 Prompt

用于评估模型回答、内容质量或任务执行结果。

典型任务：

- 评估模型回答质量
- 判断回答是否准确
- 发现遗漏、幻觉和误导
- 给出修改建议
- 生成更好的回答版本
- 判断输出是否符合要求

设计重点：

- 有明确评测维度
- 有评分标准
- 有输出格式
- 能指出具体问题
- 能区分严重问题和一般问题
- 能给出可执行的优化建议

评分重点：

- 评测维度是否合理
- 评分标准是否清楚
- 是否能定位具体问题
- 是否能解释扣分原因
- 是否能给出修改建议
- 是否能输出更好的版本

示例：

请从相关性、准确性、完整性、清晰度、安全性五个维度评估下面的模型回答，并给出评分、主要问题、修改建议和更好的回答版本。

---

### 6.4 对比型 Prompt

用于比较两个或多个 Prompt、回答、方案的优劣。

典型任务：

- 比较 Prompt A 和 Prompt B
- 判断哪个 Prompt 更适合当前任务
- 比较两个模型回答哪个更好
- 分析多个方案的优缺点
- 给出最终选择建议

设计重点：

- 对比对象明确
- 对比维度明确
- 任务背景明确
- 不能只说“哪个好”
- 要说明为什么好
- 最好给出评分表和结论

评分重点：

- 是否有清晰对比维度
- 是否分别评价每个对象
- 是否说明好坏原因
- 是否给出明确结论
- 是否给出优化建议
- 是否避免主观空泛判断

示例：

请根据目标清晰度、背景完整性、输出格式、质量标准、边界控制、可复用性六个维度，对比 Prompt A 和 Prompt B 哪个更好，并说明原因。

---

### 6.5 优化型 Prompt

用于把低质量 Prompt 改写成更标准、更可执行的 Prompt。

典型任务：

- 优化一个模糊 Prompt
- 补充角色、背景、步骤和格式
- 降低模型跑偏概率
- 让输出更稳定
- 提升 Prompt 的可复用性
- 根据模型输出问题反向优化 Prompt

设计重点：

- 先分析原 Prompt 的问题
- 再给出优化方向
- 最后输出优化后的 Prompt
- 最好解释每一处优化理由
- 保留用户真实意图，不要过度改写

评分重点：

- 是否找准原 Prompt 的问题
- 是否补齐缺失要素
- 优化后是否更清晰
- 是否更可执行
- 是否更可复用
- 是否能降低跑偏、遗漏和编造风险

示例：

请分析下面这个 Prompt 的问题，并从角色、任务目标、背景信息、执行步骤、输出格式、质量标准六个方面进行优化。  
请先说明原 Prompt 的主要问题，再给出优化后的 Prompt。

---

### 6.6 Agent Prompt

用于指导 AI Agent 进行多步骤任务执行。

典型任务：

- 判断是否需要工具
- 制定任务执行计划
- 调用工具
- 观察工具结果
- 更新任务状态
- 决定继续执行或结束
- 处理失败和不确定性

设计重点：

- 明确 Agent 角色
- 明确任务目标
- 明确可用工具
- 明确工具使用规则
- 明确执行流程
- 明确终止条件
- 明确错误处理方式
- 明确输出格式

评分重点：

- 是否能正确判断任务类型
- 是否能正确判断是否需要工具
- 是否明确工具使用顺序
- 是否包含错误处理
- 是否包含终止条件
- 是否能避免无意义循环
- 是否能说明不确定性
- 是否能在信息不足时向用户提问，而不是编造

示例：

你是一个 AI Agent 任务执行助手。  
请根据用户问题判断是否需要工具，说明判断原因，列出工具使用顺序，并给出执行方案。  
如果信息不足，不要编造，需要说明缺少什么。

---

### 6.7 安全 / Guardrails 型 Prompt

用于限制模型输出边界，降低风险内容、误导内容或不合规输出。

典型任务：

- 判断用户请求是否安全
- 拒绝不合规请求
- 提醒风险
- 避免泄露隐私
- 避免编造事实
- 控制模型不要越权操作

设计重点：

- 明确禁止内容
- 明确允许内容
- 明确拒绝方式
- 明确风险提示方式
- 明确遇到不确定信息时如何处理
- 不要只写“注意安全”，要写具体边界

评分重点：

- 是否识别风险
- 是否拒绝不合规请求
- 是否没有过度拒绝
- 是否给出安全替代方案
- 是否避免编造和误导
- 是否保护隐私和敏感信息

示例：

请判断下面用户请求是否存在安全风险。  
如果请求不合规，请说明原因并拒绝执行。  
如果可以安全回答，请在不泄露隐私、不编造事实的前提下给出帮助。

---

### 6.8 格式控制型 Prompt

用于控制模型按照固定格式输出，常见于自动化流程、数据抽取和结构化结果生成。

典型任务：

- 输出 JSON
- 输出 Markdown 表格
- 提取字段
- 按固定模板生成内容
- 生成可被程序继续处理的结构化结果

设计重点：

- 明确字段名称
- 明确字段类型
- 明确是否允许为空
- 明确输出格式
- 明确不要输出额外解释
- 必要时给出示例

评分重点：

- 是否严格符合格式
- 是否字段完整
- 是否字段类型正确
- 是否有多余内容
- 是否方便程序解析
- 是否能稳定复用

示例：

请从下面文本中提取信息，并只输出 JSON。  
字段包括：title、summary、risk_level、action_items。  
如果某个字段缺失，请填 null。  
不要输出 JSON 以外的任何内容。

## 8. 使用原则

在实际评估 Prompt 时，不要只看 Prompt 写得长不长，而要看它是否能稳定完成任务。

一个好的 Prompt 不一定很长，但应该做到：

1. 任务明确
2. 对象明确
3. 输入明确
4. 流程明确
5. 输出明确
6. 标准明确
7. 边界明确
8. 方便复用和优化

最终判断标准是：

> 这个 Prompt 是否能让模型稳定、准确、可控地完成用户真正想要的任务。

## 9. 面试表达版本

如果面试中被问到“你怎么判断一个 Prompt 好不好”，可以这样回答：

> 我会先看这个 Prompt 是否把用户需求转化成了清晰、可执行的任务说明。具体会从目标清晰度、角色设定、背景完整性、输入明确性、执行步骤、输出格式、质量标准、边界控制、可复用性和可评估性几个维度判断。
>
> 评分方式上，我会根据任务特点选择人工评分、大模型评分、规则评分或混合评分。比如格式、字段、关键词这类客观要求适合规则评分；开放式回答质量可以用大模型评分做初筛；语气、业务适配度、高风险内容则需要人工复核。真实业务里通常会用混合评分，既保证效率，也保证质量。
>
> 对于不同类型的 Prompt，评估重点也不同。学习型 Prompt 要看是否讲清楚，评测型 Prompt 要看评分维度是否合理，Agent Prompt 则要重点看工具规则、执行流程、终止条件和错误处理是否清楚。
